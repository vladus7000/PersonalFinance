# Personal Finance Assistant — план разработки MVP

**Основано на:** doc.md (ТЗ и дизайн-документ v0.1)
**Статус:** Draft v0.1
**Разработчик:** один человек, Flutter, разработка на Mac

---

# 1. Цель MVP

Дать пользователю рабочий цикл:

> настроил профиль → увидел прогноз дохода → подтвердил зарплату →
> увидел план распределения → выполнил финансовое решение (перевод/покупка) →
> прикрепил документы → увидел капитал и прогресс целей на Dashboard.

MVP должен работать **полностью локально, офлайн, без backend и без внешних интеграций**.
Критерий успеха — автор может пользоваться приложением сам минимум один полный финансовый цикл (месяц).

# 2. Функциональный состав MVP

Список сверен с разделом 2 doc.md; отмечено, что входит в MVP, а что осознанно отложено.

### Входит

| Область | Функции |
|---|---|
| Профиль | быстрый онбординг (2.1), одна основная валюта + доп. валюты |
| Доход | 1+ источник дохода, ручные правила расчёта (2.2–2.3), прогноз (2.4) |
| Подтверждение дохода | ручное подтверждение факта, без банковских интеграций (2.5) |
| Финансовый цикл | один активный цикл = месяц (2.6) |
| План распределения | правила fixed/percent/residual/priority (2.7) |
| Расходы на жизнь | только агрегат: фикс/процент/диапазон + ручной итог месяца (2.8) |
| Счета | ручное создание, ручное обновление баланса (2.9) |
| Подушка | целевая сумма/месяцы, состав по валютам (2.10) |
| Инвестиции | брокерские счета, активы, holdings, ручной ввод сделок (2.11, 2.13) |
| Стратегия | целевые доли активов, отображение отклонения — без авто-рекомендаций (2.12) |
| Финансовые решения | сущность + маршрут денег + связанные операции (2.14–2.15) |
| Документы | ручное прикрепление PDF/изображений, без распознавания (2.16, 2.17 — откладываем) |
| История | лента значимых событий (2.18) |
| Цели | базовые типы целей, ручной прогресс (2.19) |
| Планировщик/напоминания | локальные уведомления (2.21) |
| Dashboard | капитал, структура, план месяца, ближайшие события, цели, счета (2.22) |
| Аналитика | минимальный набор графиков: капитал, план/факт, структура портфеля (2.23) |

### Осознанно вне MVP

- банковские интеграции, автоопределение зарплаты (2.5, 8.6)
- автоматическое распознавание PDF/OCR (2.17, 8.10)
- FIRE-прогнозы и сценарии (2.20)
- семейный режим (8.12)
- облачная синхронизация и backend (7.10–7.11, 8.11)
- котировки активов и курсы валют через внешние API (8.8–8.9) — в MVP курс/цена вводятся вручную
- монетизация/платные функции (8.13)

# 4. Технологический стек (уже зафиксирован в doc.md §7)

- Flutter / Dart, единая кодовая база iOS + Android
- State management: Riverpod
- Навигация: go_router
- Локальная БД: SQLite + Drift, offline-first
- Простые настройки: shared_preferences
- Секьюрное хранилище: flutter_secure_storage (Keychain / Keystore)
- Модели: freezed + json_serializable
- Графики: fl_chart, за абстракцией (чтобы была возможность заменить библиотеку)
- Документы: file_picker, Share Intent/Extension, iOS PDFKit / Android PdfRenderer
- Уведомления: flutter_local_notifications
- Сеть: Dio (не используется в MVP, но подключается заранее ради консистентности со стеком)
- Тесты: flutter_test, mocktail, integration_test
- CI/CD: GitHub Actions, Fastlane → TestFlight / Play Internal Testing
- Crash reporting: Firebase Crashlytics или Sentry, без отправки сумм/документов/токенов

Архитектура — слоистая (Presentation / Application / Domain / Data / Platform),
feature-based модули как в §7.4 doc.md.

# 5. Вопросы, которые нужно закрыть до старта Phase 0

Без ответа на эти пункты архитектура и навигация будут переделываться в процессе.

1. **§8.2 Навигация** — фиксируем нижнее меню: Обзор / План / Счета / Решения / Ещё (вариант A из doc.md — согласуется с тем, что «Финансовый цикл» логично живёт внутри «Плана»).
2. **§8.5 План vs Финансовый цикл** — решить: один экран с переключателем месяцев (как описано в 3.5) или два разных экрана. Рекомендация: один экран, т.к. это сильно упрощает MVP-навигацию.
3. **§8.3 / §8.4 Документы и Инвестиции** — обе секции доступны через «Ещё» в MVP, отдельные вкладки не нужны при таком объёме функций на старте.
4. **§8.7 / §8.8 / §8.9 Календарь, курсы, цены активов** — в MVP всё вводится вручную пользователем (без внешних API), поле "источник" в модели просто хранит строку `manual`.

Эти решения приняты как рабочие допущения для плана ниже; можно пересмотреть перед Phase 2.

# 6. Этапы разработки

Оценки — ориентировочные трудозатраты одного разработчика, в неделях. Порядок фаз соответствует
зависимостям данных: сначала ядро (профиль → доход → цикл), затем капитал (счета → подушка →
инвестиции), затем связывающий слой (решения → документы), затем визуализация (dashboard/аналитика).

### Phase 0 — Инициализация проекта (≈1 неделя)
- Flutter-проект, feature-based структура каталогов
- Настройка Riverpod, go_router, Drift, freezed
- CI: lint + test на GitHub Actions
- Пустой каркас навигации (5 вкладок из §5.1 выше)
- **Готово, когда:** приложение собирается на iOS и Android, пустые экраны переключаются по нижнему меню.

### Phase 1 — Ядро данных: профиль и доход (≈2–3 недели)
- Drift-схема для `UserProfile`, `IncomeSource`, `IncomeScheduleRule`, `IncomeEvent`
- Онбординг: шаги 1–3 из §3.3 (валюта, доход, правило расчёта), быстрый старт из §2.1
- Расчёт ожидаемой выплаты (§2.3) для фиксированной и расчётной по рабочим дням зарплаты, без внешнего рабочего календаря — с ручным вводом рабочих дней/праздников на текущий месяц
- Экран прогноза зарплаты (§3.7), упрощённый (без графика курса, курс — статичное ручное поле)
- **Готово, когда:** пользователь проходит онбординг, видит рассчитанную дату и сумму следующей выплаты.

### Phase 2 — Финансовый цикл и подтверждение дохода (≈2 недели)
- `FinancialCycle`, создание текущего цикла при онбординге и автосоздание следующего при закрытии
- Экран подтверждения зарплаты (§3.8): ожидание/факт, привязка к счёту
- Экран финансового цикла (§3.6): статус, доходы, план, действия
- User flow «получение зарплаты» (§3.23) целиком
- **Готово, когда:** можно подтвердить доход, увидеть обновлённый цикл, закрыть цикл вручную (§3.26, упрощённо — без развитой аналитики план/факт).

### Phase 3 — Счета и капитал (≈1.5–2 недели)
- `Account`, `AccountBalanceSnapshot`
- Экран счетов (§3.9) и карточка счёта (§3.10)
- Расчёт общего капитала (§5.1): сумма по счетам с конвертацией по вручную заданному курсу
- **Готово, когда:** пользователь создаёт несколько счетов в разных валютах, видит капитал в основной валюте.

### Phase 4 — Подушка и цели (≈1.5 недели)
- `EmergencyFund`, `EmergencyFundComponent`, `Goal`
- Настройка подушки в онбординге (§3.3 шаг 5), экран подушки как раздел целей
- Экран целей (§3.19), прогресс, привязка к счетам
- **Готово, когда:** видно % выполнения подушки и хотя бы одной цели.

### Phase 5 — Инвестиции (≈2.5–3 недели)
- `Asset`, `Holding`, `InvestmentTransaction`
- Экран инвестиций (§3.11), карточка актива (§3.12)
- Ручной ввод покупки/продажи (§2.13), пересчёт средней цены, целевые/фактические доли (упрощённая версия §2.12 — без авто-рекомендаций, просто вывод отклонения)
- **Готово, когда:** можно добавить брокерский счёт, купить и продать актив, увидеть P&L.

### Phase 6 — Финансовые решения и маршрут денег (≈2.5 недели)
- `FinancialDecision`, `FinancialOperation`
- Мастер создания решения (§3.15, шаги 1–6)
- Карточка решения (§3.14), визуализация цепочки (§2.15, упрощённо — список шагов без графической схемы)
- User flow «инвестирование через несколько счетов» (§3.24) целиком
- **Готово, когда:** можно провести решение через несколько счетов, каждый шаг отмечается выполненным, решение завершается.

### Phase 7 — Документы (≈1.5 недели)
- `Document`, `DocumentLink`
- Ручное прикрепление файлов (file_picker + Share Extension), просмотр PDF
- Привязка документа к решению/операции/счёту/активу (без распознавания — §2.17 отложен)
- Экран документов (§3.16), без авто-полей распознавания
- **Готово, когда:** к любому решению/операции можно прикрепить документ и открыть его позже.

### Phase 8 — Dashboard и аналитика (≈2 недели)
- Экран Обзор (§3.4) целиком: капитал, структура, план месяца, ближайшие события, цели, счета, незавершённые решения
- Минимальный набор графиков аналитики (§3.20 — только капитал, план/факт, структура портфеля; остальные разделы аналитики — пост-MVP)
- **Готово, когда:** Dashboard отражает реальное состояние из всех предыдущих фаз без ручного пересчёта.

### Phase 9 — Планировщик, уведомления, безопасность (≈1.5 недели)
- `Reminder`, flutter_local_notifications
- Локальные уведомления: ожидание зарплаты, шаги решения, закрытие цикла (§2.21, §5.12)
- Биометрическая блокировка, маскирование сумм, secure storage (§5.13 — базовый набор)
- **Готово, когда:** приходят уведомления о ключевых событиях, приложение можно заблокировать биометрией.

### Phase 10 — Стабилизация и первое использование (≈1.5–2 недели)
- Обработка edge cases из §5.14, приоритетных для одиночного пользователя (частичная зарплата, отменённый перевод, неизвестный курс, незавершённый цикл)
- Unit-тесты расчётов дохода/капитала, integration-тест основных flow (§3.22–3.26)
- TestFlight / Play Internal Testing сборка
- **Готово, когда:** можно провести полный месячный цикл на реальных/тестовых данных без ручных правок базы.

**Итого: ≈19–22 недели (≈4.5–5.5 месяцев)** при работе одного разработчика с полной занятостью.
Это ориентир для последовательной разработки; часть фаз (например, 3 и 4) можно вести параллельно,
если появится второй разработчик.

# 7. Критерии готовности MVP (Definition of Done)

MVP считается готовым, когда выполнен полный цикл без обращения к базе напрямую:

1. Онбординг → создан профиль, доход, финансовый цикл.
2. Прогноз следующей зарплаты отображается корректно.
3. Зарплата подтверждена вручную, привязана к счёту, баланс обновлён.
4. План распределения показывает жизнь / подушку / инвестиции / цели.
5. Создано и выполнено финансовое решение с маршрутом через 2+ счёта и покупкой актива.
6. К решению прикреплены документы.
7. Dashboard показывает актуальный капитал, структуру, цели и незавершённые решения.
8. Цикл закрыт, создан новый.
9. Пришло хотя бы одно локальное уведомление по реальному сценарию (ожидание зарплаты или шаг решения).
10. Приложение работает полностью офлайн.

# 8. Архитектурные решения для совместимости с масс-маркет расширением

Первая реализация делается строго по скоупу разделов 1–7 этого документа — без масс-маркет
функций. Но решение о возможном выходе в масс-маркет (doc.md, раздел 10) не должно требовать
переписывания приложения через полгода. Ниже — конкретные, недорогие сейчас решения, которые
закладываются в MVP заранее, чтобы дальнейшее расширение было достройкой поверх существующей
архитектуры, а не миграцией.

Это не новые фичи MVP — это ограничения на то, *как* реализуются уже запланированные фичи.

1. **Data layer только через интерфейсы репозиториев.** Domain/Application-слой не обращается
   к Drift напрямую (уже заложено слоистой архитектурой §7.4 doc.md, но важно соблюдать строго).
   Тогда `ManualAccountRepository` в будущем заменяется/дополняется `OpenBankingAccountRepository`
   (§10.3 doc.md) без изменений в Domain и UI.

2. **Курс валют и рабочий календарь — через интерфейс с единственной MVP-реализацией.**
   `ExchangeRateSource` и `WorkCalendarSource` — абстракции с реализацией `ManualSource` в MVP.
   Расчётная логика (§2.3 doc.md) обращается к интерфейсу, а не к ручному полю напрямую — это
   не добавляет функциональности сейчас, но избавляет от переписывания calculation engine при
   подключении НБУ/внешних API позже.

3. **Банк/организация — свободный текст или конфигурируемый справочник, не enum.** Уже
   подразумевается моделью (`Account.institution: string`, §4.11) — явно зафиксировать это
   как правило при реализации UI выбора банка, чтобы не захардкодить список украинских банков
   в коде экрана.

4. **i18n-каркас с Phase 0**, даже если на старте только русский/украинский интерфейс.
   Используем `flutter_localizations`/ARB-файлы с первого экрана, а не хардкод строк — иначе
   локализация под §10.6 doc.md означает вычитку и замену текста по всему приложению.

5. **Онбординг как движок шагов, а не фиксированный wizard.** Шаги §3.3 doc.md реализуются как
   упорядоченный список компонентов с возможностью пропуска/дефолтов, а не как жёстко зашитая
   последовательность 7 экранов. Это тот же объём работы в MVP, но позволяет собрать Lite-путь
   (§10.2 doc.md) из тех же компонентов, а не писать второй онбординг с нуля.

6. **`Debt` — в схеме Drift с Phase 1, без UI в MVP.** Добавить таблицу сейчас дешевле, чем
   делать миграцию существующих баз пользователей позже, когда появится debt-first flow (§10.5).

7. **Nullable `householdId` в базовых таблицах капитала уже в MVP-схеме, не используется в UI.**
   Это позволяет избежать миграции ~10 таблиц (`Account`, `Goal`, `EmergencyFund`, `Holding` и др.)
   при добавлении семейного режима (§10.7 doc.md) — колонка добавляется один раз сейчас, а не
   ретроактивно на данных реальных пользователей.

8. **Импорт данных — через абстракцию `ImportSource` (`manual`, зарезервировано `bank_feed`,
   `pdf_ocr`).** В MVP реализован только `manual`, но `IncomeEvent`, `AccountBalanceSnapshot` и
   `Document` создаются через общий пайплайн импорта, а не через отдельный ручной путь. Тогда
   агрегация транзакций и банковских фидов (§10.3–10.4 doc.md) подключается как новый провайдер
   в тот же пайплайн, а не как параллельная система создания записей.

9. **Dashboard и другие экраны читают только через Application-провайдеры, без прямых запросов
   к БД из виджетов.** Это уже подразумевается Riverpod-архитектурой (§7.5 doc.md) — явно
   держим правило, чтобы будущий home-screen widget (§10.8 doc.md) переиспользовал ту же логику,
   а не дублировал запросы.

10. **«Агрегированные расходы на жизнь» (§2.8 doc.md) остаются отдельным агрегатом, не привязанным
    жёстко к ручному вводу.** Поле хранит сумму независимо от источника (`manual` в MVP), чтобы в
    будущем его можно было заполнять из банковского фида (§10.4 doc.md), не меняя модель
    `FinancialCycle` и не вводя полноценный transaction ledger в UI.

**Стоимость этих решений в MVP:** несколько дополнительных интерфейсов, одна лишняя таблица
(`Debt`) и одна лишняя колонка (`householdId`) — на глаз, +3–5% к объёму Phase 1 и Phase 4.
Это существенно дешевле, чем миграция реальных пользовательских данных или переписывание
calculation engine и онбординга после того, как решение о масс-маркете будет принято.

# 9. Дальнейшие шаги после MVP (см. §8.16 doc.md)

1. Личное использование 1–3 месяца, сбор проблем реального flow.
2. Автоматическое распознавание PDF (§2.17, §8.10).
3. Источники курсов валют и котировок активов (§8.8–8.9).
4. FIRE и сценарные прогнозы (§2.20).
5. Backend + синхронизация + резервные копии (§7.10–7.11).
6. Семейный режим (§8.12).
7. Решение вопроса монетизации (§8.13) и юридического позиционирования (§8.14).
